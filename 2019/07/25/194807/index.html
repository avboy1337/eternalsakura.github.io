PoC

Root Cause

在解析wasm的时候，由于83 88 88 00 00 00 ff ff前面的byte 0x44，所以它把0xffff000000888883当成f64去解析


但是在export的时候，需要将f64转化为JSValue的形式。

如上特性分析时候所说，用JSValue来表示double的时候，需要加上0x1000000000000，这样就出现了一个整数溢出。
0xffff000000888883+0x1000000000000=0x888883

于是本来导出的global[2] -> "AAAAAAA"的值是一个double值，而将其转化为JSValue，整数溢出后则是一个指针的表示形式，而这个f64的值是我们可控的，于是这个指针也就可控了。
换句话说，就有了一个double to object的类型混淆

利用思路

通过将一个double值混淆成object，我们就可以fake一个butterfly或者wasm memory这样的结构。
但是由于缺少一个leak，我们不能找到我们在哪里fake了这个结构，所以难以直接利用。

Patch

加了一个断言，并且过滤掉了我们构造的这种形式的value，返回NaN

CVE-2019-8583

pattern

这是由于wasm里表示double和将其用在js里表示double的形式不同，而对wasm进行解析的时候，只由标志0x44来判断。

find new bug

过滤相似的代码和交叉引用，发现在WasmToJS中，在使用importObject的jsFunc的时候，还是有没有加purifyNaN的地方。

构造PoC

在wasm的main中call jsFunc(arg1,arg2)的时候，在处理f64形式的arg1的时候，为了在js上下文中执行，需要将其转成JSValue，此时和上文分析的一样，存在一个整数溢出，可以将arg1溢出变成一个pointer。

举例来说，假设我们在array里fake一个object，它的地址形如0x0000 AAAA BBBB CCCC
我们可以传入0xFFFF AAAA BBBB CCCC
溢出得到的arg1就是0x0000 AAAA BBBB CCCC。
在JSValue里就代表一个指针。

漏洞利用

通过将在array里构造的fakeobj地址给leak出来，然后作为arg传入jsFunc，可以在jsFunc里将这个fakeobj实际给fake出来。
就可以在jsFunc里使用这个fakeobj来实现漏洞利用。
